# FB反映 変更用要件定義書

## 1. 結論

既存の手書き答案採点Webアプリに対して、以下の4方向の改善を検討・実装する。

| 改善領域 | 内容 | 優先度 |
|---|---|---:|
| 答案アップロード | 手書きキャンバスだけでなく、画像やPDFで答案を取り込めるようにする | 高 |
| 認識結果の確認・修正 | AIが読み取った問題・答案・数式を採点前または採点後に確認、手動修正できるようにする | 高 |
| 段階的な採点フロー | 「テキスト化 → 比較 → 採点」の流れを明示し、AIにも段階的に処理させる | 高 |
| 赤ペン重ね表示 | 答案画像上にミス位置、注意点、コメントをオーバーレイ表示する | 中 |
| 学習者別ミス分析 | 過去の採点履歴から苦手傾向や繰り返すミスを分析する | 中 |

本変更の狙いは、単なる「AIに画像を投げて採点するアプリ」ではなく、認識の透明性、採点への納得感、学習履歴の蓄積、画像上の直感的なフィードバックを備えた学習支援アプリへ進化させることである。

## 2. 背景

ユーザーFBでは、現状の強みとして以下が挙げられた。

| 現状の強み | 内容 |
|---|---|
| ブラウザ上で直接手書きできる | iPad等でそのまま答案を書ける |
| API連携が動いている | 手書き答案がAI採点までつながっている |
| UIが直感的 | 画面がすっきりしていて使いやすそう |
| 文字認識が強い | 潰れた字でもある程度認識して採点できる |

一方で、今後の改善案として以下が挙げられた。

| 改善案 | 期待される価値 |
|---|---|
| ノートアプリ等で書いた答案画像・PDFのアップロード | 普段の学習フローに乗せやすい |
| 過去のミス分析 | 通常のAIチャットとの差別化 |
| AIの読み取り結果を表示・修正 | 認識ミスによる不公平な減点を防ぐ |
| 段階的な採点 | 精度と納得感の向上 |
| 答案画像上への赤ペン表示 | どこを直せばよいか直感的に分かる |

## 3. 変更方針

### 3.1 既存機能を壊さない

既存の直接手書きキャンバス、問題画像、模範解答画像、Gemini採点、履歴保存、利用量メーターは継続する。

### 3.2 採点前に「AIの読み取り」を見える化する

AIの採点結果だけを表示するのではなく、採点の根拠となる読み取り内容をユーザーが確認できるようにする。  
認識ミスがある場合は、ユーザーが修正してから採点できるようにする。

### 3.3 AIには画像加工ではなく構造化データを出させる

赤ペン表示は、AIに画像そのものを加工させるのではなく、以下のような構造化データを出力させ、アプリ側で描画する。

```json
{
  "annotations": [
    {
      "type": "mistake",
      "x": 0.42,
      "y": 0.58,
      "width": 0.18,
      "height": 0.08,
      "message": "符号が逆になっています",
      "severity": "medium"
    }
  ]
}
```

座標は画像サイズに依存しないよう、0.0から1.0の相対座標を基本とする。

## 4. 対象範囲

### 4.1 今回追加する主要機能

| 機能 | 内容 | 優先度 |
|---|---|---:|
| 答案画像アップロード | 手書きキャンバスの代わりに画像を答案として取り込む | 高 |
| 答案PDFアップロード | PDFの1ページ目または選択ページを答案画像化して採点する | 中 |
| 認識結果レビュー | AIが読み取った問題、答案、数式を画面に表示する | 高 |
| 認識結果編集 | 読み取りテキストやLaTeXを手動修正する | 高 |
| 段階採点API | OCR、比較、採点を分けた処理として実行できるようにする | 高 |
| 赤ペン注釈 | 答案画像上に赤枠、マーカー、短いコメントを重ねる | 中 |
| ミス履歴分析 | 過去の採点結果からよくあるミスを集計する | 中 |
| 採点厳しさ設定 | 甘め、標準、厳しめなどの採点方針を選べる | 中 |

### 4.2 対象外

| 項目 | 理由 |
|---|---|
| AIによる画像ファイル自体の直接加工 | 実装が重く、まずはオーバーレイ描画で十分な価値があるため |
| 複数ページPDFの一括採点 | 初期実装では複雑になりすぎるため |
| ユーザーアカウント管理 | まずは端末内履歴でミス分析を行うため |
| サーバー側の長期学習DB | 個人情報・コスト・設計負荷が高いため |

## 5. 画面構成変更

### 5.1 入力ワークスペース

既存の入力タブに、答案の取り込み方法を追加する。

```text
--------------------------------------------------+
| [答案] [問題・模範解答] [認識結果] [赤ペン]       |
+--------------------------------------------------+
| 答案タブ                                         |
| [手書き] [画像アップロード] [PDFアップロード]     |
|                                                  |
| 手書きキャンバス または 取り込んだ答案画像        |
+--------------------------------------------------+
```

### 5.2 認識結果タブ

AIが読み取った内容を表示し、編集できるようにする。

| 表示項目 | 編集可否 | 内容 |
|---|---:|---|
| 読み取り問題 | 可 | 問題文・問題画像から読み取った内容 |
| 読み取り答案 | 可 | 手書き答案またはアップロード答案から読み取った内容 |
| 読み取り数式LaTeX | 可 | 主要な数式 |
| 認識信頼度 | 不可 | high / medium / low |
| 認識メモ | 不可 | AIが不確実と判断した箇所 |

### 5.3 赤ペンタブ

答案画像またはキャンバス画像を背景にして、AIの注釈を重ねる。

| 表示要素 | 内容 |
|---|---|
| 赤枠 | 誤りのある箇所を囲む |
| 下線 | 注意すべき式や記述を示す |
| コメントラベル | 短い改善コメント |
| 注釈一覧 | 画面外や小さい注釈も一覧で確認できる |
| 表示切替 | ミス、良い点、注意点の表示オンオフ |

## 6. 採点フロー変更

### 6.1 現状フロー

```text
問題・模範解答・答案画像
  ↓
Geminiへ一括送信
  ↓
採点結果を表示
```

### 6.2 変更後フロー

```text
問題・模範解答・答案画像
  ↓
Step 1: AI OCR / 数式読み取り
  ↓
認識結果を表示
  ↓
ユーザーが必要に応じて修正
  ↓
Step 2: 模範解答との比較
  ↓
Step 3: 採点・フィードバック・赤ペン注釈生成
  ↓
結果表示・履歴保存
```

### 6.3 操作モード

| モード | 内容 | 初期採用 |
|---|---|---:|
| すぐ採点 | OCR、比較、採点を連続実行する | 採用 |
| 確認して採点 | OCR結果を確認・修正してから採点する | 採用 |
| OCRだけ | 採点せず、読み取り結果だけ確認する | 任意 |

初期UIでは「すぐ採点」を簡単に使えるようにしつつ、認識ミスが不安なユーザー向けに「確認して採点」を用意する。

## 7. 答案アップロード要件

### 7.1 画像アップロード

| 項目 | 内容 |
|---|---|
| 入力方法 | `<input type="file" accept="image/*">` |
| 対象 | PNG、JPEG、HEIC変換後画像、スクリーンショット |
| 用途 | 手書きキャンバスの代替答案として採点 |
| プレビュー | 答案タブ内に表示 |
| 編集 | 初期版ではトリミングなし。将来追加候補 |
| 保存 | IndexedDBにData URLまたはBlobとして保存 |

### 7.2 PDFアップロード

| 項目 | 内容 |
|---|---|
| 入力方法 | `<input type="file" accept="application/pdf">` |
| 対象 | ノートアプリやスキャンアプリから出力したPDF |
| 初期対応 | 1ページ目を画像化して採点 |
| 将来対応 | ページ選択、複数ページ答案 |
| 注意 | PDFレンダリングライブラリの追加が必要 |

### 7.3 答案入力ソース

| ソース | 内容 |
|---|---|
| canvas | アプリ上で直接手書きした答案 |
| uploadedImage | 画像として取り込んだ答案 |
| uploadedPdfPage | PDFから画像化した答案 |

採点時には、どのソースであっても最終的に `answerImage` としてAPIへ渡す。

## 8. 認識結果修正要件

### 8.1 修正対象

| 項目 | 内容 |
|---|---|
| recognizedQuestion | AIが読み取った問題 |
| recognizedAnswer | AIが読み取った答案 |
| recognizedLatex | AIが読み取った主要数式 |
| userCorrectedQuestion | ユーザー修正後の問題 |
| userCorrectedAnswer | ユーザー修正後の答案 |
| userCorrectedLatex | ユーザー修正後のLaTeX |

### 8.2 採点時の扱い

ユーザー修正がある場合、採点APIはAIの元認識結果ではなく、ユーザー修正後の内容を優先して使用する。

| 状況 | 採点に使う内容 |
|---|---|
| 修正なし | AI認識結果 |
| 修正あり | ユーザー修正結果 |
| 認識信頼度low | 確認して採点を推奨 |

### 8.3 UI要件

| 項目 | 内容 |
|---|---|
| 編集UI | textareaまたは数式編集欄 |
| 差分表示 | 初期版では不要。将来候補 |
| リセット | AI認識結果に戻すボタン |
| 採点ボタン | 「この内容で採点」 |

## 9. 赤ペン注釈要件

### 9.1 AI出力

Geminiには採点結果と併せて、答案画像上に重ねる注釈データをJSONで返すよう指示する。

```json
{
  "annotations": [
    {
      "id": "ann-1",
      "type": "mistake",
      "target": "answer",
      "x": 0.35,
      "y": 0.48,
      "width": 0.22,
      "height": 0.07,
      "message": "計算途中で符号が逆になっています",
      "severity": "high",
      "relatedMistakeIndex": 0
    }
  ]
}
```

### 9.2 注釈種別

| type | 表示 | 内容 |
|---|---|---|
| mistake | 赤枠 | 明確な誤り |
| warning | オレンジ下線 | 誤りの可能性、説明不足 |
| good | 緑チェック | 良い途中式、正しい方針 |
| note | 青ラベル | 補足コメント |

### 9.3 座標仕様

| 項目 | 内容 |
|---|---|
| x, y | 左上を基準にした相対座標 |
| width, height | 相対サイズ |
| 範囲 | 0.0から1.0 |
| 対象画像 | 採点に使った答案画像 |
| 表示 | CSSまたはCanvasでオーバーレイ |

### 9.4 制約

AIの座標推定は完全には正確でないため、以下のUIを用意する。

| 制約 | 対応 |
|---|---|
| 注釈位置がずれる | 注釈一覧にも同じコメントを表示 |
| 座標が不正 | 該当注釈は一覧のみ表示 |
| 画像リサイズでずれる | 相対座標で再計算 |
| 信頼度が低い | 「位置は目安」と表示 |

## 10. ミス分析要件

### 10.1 目的

通常のAIチャットとの差別化として、過去の採点履歴からユーザーの苦手傾向を表示する。

### 10.2 分析対象

| データ | 内容 |
|---|---|
| mistakes | 採点ごとの誤り |
| improvements | 改善ポイント |
| recognizedLatex | 間違えた数式 |
| problemTags | 問題の単元や種類 |
| scoreRate | 得点率 |
| createdAt | 採点日時 |

### 10.3 表示内容

| 表示 | 内容 |
|---|---|
| よくあるミス | 符号ミス、計算ミス、条件漏れなど |
| 苦手単元 | 方程式、微分、積分など |
| 改善アドバイス | 次に意識すべきポイント |
| 推移 | 得点率やミス数の変化 |
| 復習候補 | 似たミスをした過去答案 |

### 10.4 初期実装方針

初期版では、AIにミスカテゴリを返させ、IndexedDB内の履歴から端末内で集計する。

```json
{
  "mistakeCategories": [
    "sign_error",
    "calculation_error",
    "missing_explanation"
  ]
}
```

## 11. 採点厳しさ設定

### 11.1 設定項目

| 設定 | 内容 |
|---|---|
| 甘め | 考え方が合っていれば部分点を多めにする |
| 標準 | 一般的な学校採点に近いバランス |
| 厳しめ | 記述不足や途中式不足も減点する |
| カスタム | 採点基準テキストで細かく指定 |

### 11.2 API送信

採点APIへ `gradingStrictness` を送信する。

```json
{
  "gradingStrictness": "standard"
}
```

### 12.1 採点履歴への追加項目

| 項目名 | 型 | 内容 |
|---|---|---|
| answerInputSource | 文字列 | canvas / uploadedImage / uploadedPdfPage |
| uploadedAnswerImageDataUrl | 文字列 | 取り込んだ答案画像 |
| uploadedAnswerFileName | 文字列 | 元ファイル名 |
| uploadedAnswerMimeType | 文字列 | MIME type |
| recognizedQuestionDraft | 文字列 | AI読み取り問題 |
| recognizedAnswerDraft | 文字列 | AI読み取り答案 |
| recognizedLatexDraft | 文字列 | AI読み取りLaTeX |
| userCorrectedQuestion | 文字列 | ユーザー修正問題 |
| userCorrectedAnswer | 文字列 | ユーザー修正答案 |
| userCorrectedLatex | 文字列 | ユーザー修正LaTeX |

| mistakeCategories | 配列 | ミス分類 |
| gradingStrictness | 文字列 | 採点の厳しさ |

## 13. API変更要件

### 13.1 API分割案

| API | 役割 | 優先度 |
|---|---|---:|
| `/api/recognize` | 問題・答案画像を読み取り、構造化テキストを返す | 高 |
| `/api/grade` | 認識結果、模範解答、採点基準から採点する | 高 |
| `/api/analyze-history` | 履歴からミス傾向を分析する | 低 |

### 13.2 既存API維持案

初期実装では `/api/grade` のまま以下の `mode` を追加してもよい。

| mode | 内容 |
|---|---|
| recognize | 読み取りのみ |
| grade | 採点のみ |
| full | 読み取りから採点まで一括 |

将来的な見通しはAPI分割の方がよいが、実装コストを抑えるなら `mode` 追加から始める。

## 14. 受け入れ条件

| No | 条件 |
|---:|---|
| 1 | 手書きキャンバス以外に、答案画像をアップロードできる |
| 2 | アップロードした答案画像をプレビューできる |
| 3 | アップロードした答案画像を採点に使える |
| 4 | PDFをアップロードした場合、少なくとも1ページ目を答案として扱える |
| 5 | AIが読み取った問題・答案・数式を表示できる |
| 6 | 読み取り結果をユーザーが修正できる |
| 7 | 修正後の読み取り結果を使って採点できる |
| 8 | 「すぐ採点」と「確認して採点」を選べる |
| 9 | 採点結果にミス位置の注釈データが含まれる |
| 10 | 答案画像上に赤枠やコメントを重ねて表示できる |
| 11 | 注釈位置が不正な場合でも注釈一覧で内容を確認できる |
| 12 | 採点履歴にミスカテゴリを保存できる |
| 13 | 過去履歴からよくあるミスを表示できる |
| 14 | 採点の厳しさを選択できる |
| 15 | 既存の直接手書き採点フローが引き続き使える |

## 15. MVP範囲

### 15.1 まず入れる

| 機能 | 理由 |
|---|---|
| 答案画像アップロード | 利用シーンが大きく広がる |
| 認識結果表示 | 採点への納得感が上がる |
| 認識結果編集 | AI認識ミスによるストレスを減らせる |
| 確認して採点モード | 精度と安心感を両立できる |
| 採点厳しさ設定 | 実装が比較的軽く、価値が分かりやすい |

### 15.2 次に入れる

| 機能 | 理由 |
|---|---|
| 赤ペン注釈 | 差別化要素として強いが、座標精度の検証が必要 |
| ミスカテゴリ保存 | 履歴分析の土台になる |
| よくあるミス表示 | アプリ独自価値を作れる |

### 15.3 後回し

| 機能 | 理由 |
|---|---|
| PDF複数ページ対応 | UIとデータ構造が複雑になる |
| サーバー側ユーザー別分析 | アカウント設計が必要 |
| 注釈の手動移動・編集 | 初期版では一覧表示で補える |

## 16. 未確定事項

| 項目 | 状態 |
|---|---|
| PDFレンダリングライブラリ | 要選定 |
| HEIC画像の扱い | iPad Safariでの挙動確認が必要 |
| 赤ペン座標の精度 | 実データで検証が必要 |
| 認識結果修正UI | textarea中心か、数式エディタを入れるか未確定 |
| API分割 | `/api/recognize` 新設か `/api/grade` mode追加か未確定 |
| ミスカテゴリ体系 | 初期カテゴリを設計する必要あり |
| 履歴分析の粒度 | 端末内集計で十分か要検討 |

## 17. 推奨実装順

1. 答案画像アップロードを追加する
2. 採点時の答案ソースを `canvas / uploadedImage` で切り替えられるようにする
3. AI読み取り結果を採点結果とは別に保存・表示する
4. 読み取り結果を編集してから採点できるモードを追加する
5. 採点厳しさ設定を追加する
6. AI出力に赤ペン注釈JSONを追加する
7. 答案画像上に注釈オーバーレイを表示する
8. ミスカテゴリを履歴保存する
9. 履歴からよくあるミスを集計表示する
10. PDFアップロード対応を追加する

## 18. 確信度

| 項目 | 確信度 | 理由 |
|---|---|---|
| 答案画像アップロード | 高 | 既存の画像アップロード実装を応用できる |
| 認識結果表示 | 高 | 既にGemini返却JSONに読み取り結果がある |
| 認識結果編集 | 高 | フロントエンドの状態管理とAPI送信項目追加で実現できる |
| 段階採点 | 中 | API設計変更とプロンプト分割が必要 |
| 赤ペン注釈 | 中 | オーバーレイ表示は可能だがAI座標精度の検証が必要 |
| PDFアップロード | 中 | PDFレンダリング処理の追加が必要 |
| ミス分析 | 中 | 履歴データは使えるが分類の安定性はプロンプトに依存する |
